# AGENTS.md

## Project Overview

**behtree** is a Go library for constructing, validating, interpreting, and simulating behaviour trees. It uses the PA-BT algorithm (Colledanchise & Ogren) to automatically build provably correct trees from action definitions with pre/postconditions. LLMs participate by selecting and grounding actions — essentially function-calling — rather than generating tree structure directly.

See `HISTORY.md` for the epoch 1 approach (LLM-generated trees) and why we pivoted.

## Architecture

```
condition.go         Condition types: template, resolved, StateRef
pabt.go              PA-BT algorithm: BuildTree, GroundAction, action grounding
document.go          Parse/load JSON documents
environment.go       Type system: objects, interfaces, action definitions
  ├─ registry.go     Runtime handler registry (Go functions per action)
  ├─ state.go        Runtime state (object → field → value)
  └─ validate.go     Static validation of environment + tree structure
behtree.go           Core types: Node, NodeType, Status, Params
interpreter.go       Tree-walking tick interpreter
  └─ runtasktree.go  Dynamic subtree execution (RunTaskTree)
harness.go           Exhaustive simulation (all outcome combinations)
benchmark.go         LLM benchmarking framework (PA-BT pipeline)
print.go             ASCII tree pretty-printer
```

### Key types

| Type | File | Purpose |
|------|------|---------|
| `Condition` | condition.go | Template condition with `$param` references |
| `ResolvedCondition` | condition.go | Grounded condition (literal or StateRef values) |
| `GroundedAction` | pabt.go | Action with resolved pre/postconditions |
| `BuildResult` | pabt.go | PA-BT output: tree + auto-registered condition handlers |
| `Node` | behtree.go | Tree node with type, name, params, children |
| `ActionDef` | environment.go | Action definition with params and condition templates |
| `Environment` | environment.go | Merged definitions: objects, interfaces, actions, goal |
| `ActionRegistry` | registry.go | Maps action/condition names → Go handler functions |
| `State` | state.go | Runtime variable store (object.field = value) |
| `Interpreter` | interpreter.go | Ticks a tree against registry + state |
| `SimulationHarness` | harness.go | Runs all outcome permutations for a tree |

### Node types

- **Control**: `Sequence` (all-must-succeed), `Fallback` (first-success-wins)
- **Leaf**: `Condition` (read-only, auto-generated by PA-BT), `Action` (side effects, may RUNNING)

No memory variants — nodes always re-evaluate from the start on each tick.

### PA-BT Pipeline

1. Define actions with pre/postconditions in JSON (template conditions use `$param` refs)
2. LLM selects actions and grounds parameters (or grind all combos for testing)
3. `GroundAction()` resolves `$param` refs to literal values / StateRefs
4. `BuildTree()` constructs tree via PA-BT algorithm
5. Condition handlers are auto-registered in the returned registry
6. Tree is executed via interpreter + action handlers

### Condition System

Template conditions use `$` prefix for parameter references:
- `$param` in object/value → resolved to `params[param]` (literal)
- `$param.field` in value → resolved to `StateRef{Object: params[param], Field: field}` (runtime lookup)

## Code Conventions

- **Go 1.25**, single-package `behtree` (no sub-packages)
- JSON is the primary serialization format; `Node` has custom `UnmarshalJSON`
- Handler signature: `func(Params, *State, OutcomeRequest) HandlerResult`
  - `OutcomeRequest` enables simulation: handlers check if a requested outcome is compatible with current state
- `Environment.Merge()` combines multiple document definitions additively
- Validation is static and separate from interpretation — call `Validate(env)` before ticking
- State is a flat two-level map: `object name → field name → value`
- Actions use `ActionDef` (aliased as `BehaviourDef` for backward compat)
- Registry uses `ActionRegistry` (aliased as `BehaviourRegistry` for backward compat)
- Condition nodes are auto-generated by PA-BT — users define only Action behaviours

## Design Decisions

- **PA-BT over LLM tree generation** — LLMs select actions (function-calling), PA-BT builds correct trees
- **No memory nodes** — all control nodes re-evaluate every child on each tick
- **Condition nodes are auto-generated** — PA-BT creates them from action pre/postconditions
- **StateRef for dynamic conditions** — `$param.field` resolves to runtime state lookups
- **Simulation uses outcome requests** — the harness asks handlers to produce specific outcomes
- **Dynamic subtrees** — `RunTaskTree` executes PA-BT inner trees within PA-BT outer trees
- **Ephemeral state fields** — `observed` and `idle` fields are reset at tick start by the interpreter, forcing re-observation of external state and re-evaluation of derived conditions
- **Generic Observe action** — a single `Observe(target)` action re-syncs internal state with reality; the handler dispatches to target-specific observers and can internally cache or use cheaper heuristics
- **Worst-case planning state** — trees must be built from worst-case initial state so PA-BT expands all fallback branches; runtime state may differ
