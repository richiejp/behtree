## Project Overview

**behtree** is a Go library for constructing, validating, interpreting, and simulating behaviour trees. It uses the PA-BT algorithm (Colledanchise & Ogren) to automatically build provably correct trees from action definitions with pre/postconditions. LLMs participate by selecting and grounding actions â€” essentially function-calling â€” rather than generating tree structure directly.

See `HISTORY.md` for the epoch 1 approach (LLM-generated trees) and why we pivoted.

### Node types

- **Control**: `Sequence` (all-must-succeed), `Fallback` (first-success-wins)
- **Leaf**: `Condition` (read-only, auto-generated by PA-BT), `Action` (side effects, may RUNNING)

No memory variants â€” nodes always re-evaluate from the start on each tick.

### PA-BT Pipeline

1. Define actions with pre/postconditions in JSON (template conditions use `$param` refs)
2. LLM selects actions and grounds parameters (or grind all combos for testing)
3. `GroundAction()` resolves `$param` refs to literal values / StateRefs
4. `BuildTree()` constructs tree via PA-BT algorithm
5. Condition handlers are auto-registered in the returned registry
6. Tree is executed via interpreter + action handlers

### Condition System

Template conditions use `$` prefix for parameter references:
- `$param` in object/value â†’ resolved to `params[param]` (literal)
- `$param.field` in value â†’ resolved to `StateRef{Object: params[param], Field: field}` (runtime lookup)

## Code Conventions

- **Go 1.25**, single-package `behtree` (no sub-packages)
- JSON is the primary serialization format; `Node` has custom `UnmarshalJSON`
- Handler signature: `func(Params, *State, OutcomeRequest) HandlerResult`
  - `OutcomeRequest` enables simulation: handlers check if a requested outcome is compatible with current state
- `Environment.Merge()` combines multiple document definitions additively
- Validation is static and separate from interpretation â€” call `Validate(env)` before ticking
- State is a flat two-level map: `object name â†’ field name â†’ value`
- Actions use `ActionDef` (aliased as `BehaviourDef` for backward compat)
- Registry uses `ActionRegistry` (aliased as `BehaviourRegistry` for backward compat)
- Condition nodes are auto-generated by PA-BT â€” users define only Action behaviours

## Design Decisions

- **PA-BT over LLM tree generation** â€” LLMs select actions (function-calling), PA-BT builds correct trees
- **No memory nodes** â€” all control nodes re-evaluate every child on each tick
- **Condition nodes are auto-generated** â€” PA-BT creates them from action pre/postconditions
- **StateRef for dynamic conditions** â€” `$param.field` resolves to runtime state lookups
- **Simulation uses outcome requests** â€” the harness asks handlers to produce specific outcomes
- **Dynamic subtrees** â€” `RunTaskTree` enables LLM-generated inner trees composed within fixed outer trees
